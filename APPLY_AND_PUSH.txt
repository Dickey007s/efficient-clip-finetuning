Sync the new headline runs to GitHub
=====================================

WHY THIS FILE EXISTS
--------------------
All the data work is done and packaged into a ready-made commit:

    3b7c7a0  data: add converged 4-shot M4 and 16-shot M5 headline runs

It sits on top of your existing local commit f12fda4 (the checkpoint feature,
which is also not yet on GitHub). The commit is stored in the bundle file
"clip_sync.bundle" in this folder.

The actual `git push` could NOT be done automatically, because the assistant
sandbox has no GitHub write credentials (it can only read public repos). The
push has to run from your machine, where your GitHub login lives.

WHAT THE COMMIT CONTAINS
------------------------
- outputs/main/m5/m5_16shot_lora40_r4_a8/   (16-shot M5 = 81.67%, + coop/lora .pt)
- outputs/main/m4/m4_4shot_r4_a8_{40,50,60}ep_seed42/  (4-shot M4 sweep -> 65.38%,
  final 60ep .pt only)
- results/tables/*  regenerated
- README.md, README.zh-CN.md  updated (low-shot M4-vs-M5 matrix now complete)
- src/curate_outputs.py  registers the two runs

PUSH IT (run in this folder, on Windows where git works normally)
-----------------------------------------------------------------
    git fetch clip_sync.bundle main
    git reset --hard FETCH_HEAD
    git push origin main

`git reset --hard` is safe here: every change it discards is already inside the
commit. The only thing it reverts is the small uncommitted edit in
src/train_m5d.py (the --coop_ckpt argument). If you want to keep that, run
`git stash push -- src/train_m5d.py` before the reset and `git stash pop` after.

CLEANUP (optional, Windows can delete what the sandbox could not)
----------------------------------------------------------------
The sandbox drive mount blocks file deletion, so a few harmless leftovers
remain that you can remove:
    del .git\index.lock.stale
    del .git\index.lock.REMOVE_ME
    git gc            (prunes the stray .git/objects/**/tmp_obj_* files)
    del clip_sync.bundle  APPLY_AND_PUSH.txt   (once you've pushed)
